# AI 協作驗證規範 (AI Collaboration & Verification Standard)

> **要解決的痛點：** 工程師最苦惱的不是 AI 不會做事，而是
> **「我怎麼知道 AI 講的是對的？我怎麼知道它的運作流程是對的？」**
>
> 本規範把「驗證 AI」當成一個工程問題來解，適用於 **IC 設計 / DV 驗證 / 軟體開發** 三種流程。
>
> 版本 v1 · 適用對象：任何 AI 協作（Claude / GPT / Gemini / 自架 agent 皆通用）

---

## 心法一句話：把 AI 的輸出當成 DUT

你不會因為 RTL「看起來對」就 tape out。你會 **cover 它、assert 它、用 scoreboard 比對它**。
**對 AI 的每一句話、每一段流程，用完全一樣的紀律。**

| DV 概念 | 對應到 AI 協作 |
|---|---|
| DUT（待驗證設計） | AI 的每個輸出（答案、程式碼、數據、計劃） |
| Spec / Golden Model | 真實來源（原始文件、工具報告、規格書） |
| Assertion (SVA) | 「可執行的主張」——把 AI 的『應該會』變成能被檢查的條件 |
| Scoreboard | 獨立的對照器（第二來源、第二工具、人工抽查） |
| Coverage | 「真的驗到了嗎」的證據，不是「我覺得驗過了」 |
| Constrained-random / corner case | 挑戰式測試：專找它會錯的地方，而不是讓它過 |
| Regression + seed | 可重現：同樣輸入要能重跑出同樣結果 |

> **核心原則：不是「信任 AI」，而是「讓 AI 的一切可稽核」。**
> 可以信任的前提是可被獨立檢查；**不可驗證的主張，無論多流暢，價值等於零（甚至為負，因為它最危險）。**

## 七條鐵則 (The 7 Rules)

### R1. 每個主張都要帶「來源 + 信心標籤」(Claim → Source → Confidence)
任何事實型陳述後面必須掛標籤：

- `[VERIFIED]` — 已用工具/原始文件查證（附來源連結或報告路徑）
- `[ESTIMATE]` — 推估、模型、判斷（非事實）
- `[STALE]` — 可能過期（附時間）
- `[ASSUMPTION]` — 假設（必須能被質疑）
- **無標籤 = 視同 `[UNVERIFIED]` = 不可採信**

> 規則：**沒有來源的數字，當它不存在。**

### R2. 事實與判斷必須分開
「BTC 現價 $61,900 `[VERIFIED]`」是事實；「BTC 兩年後 $130k」是判斷。
**預測、情境、機率永遠是 `[ESTIMATE]`，不准混進事實表。** 工程上：「這條 path timing 是 -50ps `[VERIFIED-STA]`」vs「我覺得改完應該會 met `[ESTIMATE]`」是兩件事。

### R3. 交叉比對 ≥ 2 個獨立來源
單一來源 = 未驗證。
- 一般：第二來源 / 第二工具。
- IC/DV：**spec ↔ RTL ↔ testbench 三方交叉**，任何兩者不一致＝有 bug（不一定是 RTL 錯，可能是 TB 或 spec 錯）。
- 自動資料管線：**資料欄位是契約，供應商是可替換元件**。指定來源失敗時，改用其他獨立來源；只要來源數、新鮮度與跨源價差仍通過，該欄位維持通過，來源故障另記為 incident，不得誤報成資料失敗。
- 只剩單一來源、時間戳逾時或跨源價差超標時，必須維持未知或 fail closed；禁止沿用舊值、補零或把單源值偽裝成已驗證。

### R4. 一切可重現 (Reproducibility)
要求 AI 給出**確切指令、版本、輸入、seed**，讓你能自己重跑。
「我跑過了」不算數，「這是指令你自己跑」才算。

### R5. 流程要顯性化 (Make the process visible)
AI 必須交出**它怎麼拆題、做了哪些步驟、在哪裡做了取捨、改了什麼**——不是只給結論。
看不見過程，就無法信任結果。具體技巧見下方〈讓過程可見的 6 個技巧〉。

### R6. 挑戰，不附和 (Adversarial, not agreeable)
預設 AI（和你自己）都會犯「自信的錯」。
- 要它**論證自己為什麼會錯**（red team）。
- 主動找**內部矛盾**（例：兩份資料／spec 前後段／RTL 與註解互打臉）。
- 寫測試是為了**抓 bug**，不是為了讓它 pass。

### R7. 誠實更新並留痕 (Honest update with a log)
當新證據推翻舊主張（即使是 AI 自己或你先前的結論），**明確更正並記錄**「原本說 X，因為 Y 改成 Z」。
無紀錄的更新 = 無法追溯 = 不可信。

## 標準流程 (6 Phases) — 每一階段都有「人的關卡 (Gate)」

> AI 負責「並行展開 + 收斂」，人負責「定義對錯 + 拍板」。每個 Phase 結束，人要過一道 Gate 才放行。

| Phase | AI 做什麼 | 人的 Gate（你要檢查的） |
|---|---|---|
| **0. 理解既有狀態** | 先讀懂現有的 code / spec / 半成品，**不要從零重來** | 它有沒有真的讀懂？要它複述現況與已知缺口 |
| **1. 拆解 + 並行** | 把問題拆成**互相獨立**的子任務，各自帶「輸出契約」 | 拆解合理嗎？子任務有重疊或漏洞嗎？ |
| **2. 查證 + 糾錯** | 每個子任務查證來源、交叉比對、**主動標記前人/自己的錯** | 抽查幾個 `[VERIFIED]` 是否真的查證；錯誤有沒有被抓出 |
| **3. 綜合 + 判斷** | 把事實收斂成結論，**事實與判斷分標** | 結論是否從事實推得？判斷有沒有偽裝成事實？ |
| **4. 對抗審查** | 自我紅隊：找矛盾、找 corner case、列出最不確定的點 | 它敢不敢打自己臉？最弱的三點是什麼？ |
| **5. 顯性交付** | 交出：結論 + 過程 + 信心標籤 + **「我沒驗證的東西」** | 過程可稽核嗎？未知數有沒有誠實列出？ |

## 讓過程可見的 6 個技巧（這次研究實際用的手法）

1. **信心標籤滿版** — 每個數字後面 `[VERIFIED]/[ESTIMATE]/[STALE]`，一眼分辨硬數據與推估。
2. **「我沒能驗證的東西」專區** — 強制 AI 列出未知數與資料衝突，而不是默默蓋過去。**這一段往往比結論更有價值。**
3. **糾錯對照表** — 「原本說 X / 查證實際 Y / 判定 ✗」並排，讓修正過程被看見（這次抓出 4 個前版錯誤）。
4. **迭代記錄 (iteration log)** — 主席→質疑→修正→再質疑→批准，把來回辯論留痕。
5. **事實表 vs 判斷區分版面** — 兩者用不同區塊/顏色，避免「預測偷渡成事實」。
6. **可重現的工具足跡** — 用了什麼工具、什麼指令、什麼來源連結，全部附上，你能自己重跑。

> 反向自查：如果一份 AI 交付物**沒有來源、沒有信心標籤、沒有「我不確定什麼」、沒有過程**——它就是一個你無法驗證的黑箱，應該退回。

## 驗證工具箱：怎麼實際抓 AI 的錯

對 AI 下這幾道指令，逼出可驗證性：

- **「給我能跑的證據，不要給我結論」** → SW：測試/實際執行；IC：工具報告；DV：sim log + coverage。
- **「列出你最不確定的 3 點，以及一旦錯了會怎樣」** → 逼它暴露弱點。
- **「列出你完全沒驗證的東西」** → 把未知數攤開。
- **「論證這個結論為什麼可能是錯的」** → red team 自己。
- **「這個主張要用什麼工具/實驗才能證偽？」** → 沒有證偽方法的主張＝信仰，不是工程。
- **抽樣深查** → 隨機挑 2–3 個它說 `[VERIFIED]` 的，自己追到底；**若這幾個站不住，整份退回。**
- **小步交付** → 要求可逐步驗證的小增量，拒絕「大爆炸式」一次給一大包無法分段檢查的東西。

## 領域對應

### 🔲 IC 設計流程

| AI 的主張 | 不准用 AI 的話當證據，要用這個 oracle |
|---|---|
| 「這段 RTL 行為是…」 | **模擬波形 / lint / formal**；改完跑 **LEC 等價驗證** |
| 「timing 會 met / 這條 path 是 X」 | **STA 報告**（指定 corner/mode），不是 AI 推測 |
| 「面積/功耗大概…」 | synth / power report |
| 「這個 CDC/RDC 沒問題」 | **CDC 工具**，AI 不准背書跨時脈域 |
| 「constraint(SDC) 這樣寫對」 | STA + 報告確認 clock/IO/exception 真的生效 |

- **鐵則：AI 改的每一行 RTL，進 repo 前必須過 lint + sim（+ 關鍵路徑 formal/LEC）。**
- 把 R3 落地：**spec ↔ RTL ↔ TB 三方比對**，AI 說「RTL 對」時反問「對照哪一條 spec？哪個測試覆蓋到？」

### 🔳 DV 驗證流程（你的主場，講最細）

> DV 本身就是「不相信 DUT、用獨立機制證明它對」的學問——把同一套紀律套在 AI 身上即可。

- **AI 寫的 testbench/sequence/assertion 一律先當「未驗證」** → 對照 spec 逐條確認，**不可假設它正確**。AI 寫的 checker 本身也是 DUT。
- **Coverage 是 ground truth** → 別信 AI 說「測過了」；看 **functional coverage + code coverage** 的數字。沒覆蓋到 = 沒測到。
- **把 AI 的『應該會』變成 SVA** → 它說「reset 後該清零」，就寫成 `assert`，讓模擬去證明，而不是嘴上保證。
- **Scoreboard / golden model = AI 主張的獨立裁判** → 結果對不對，交給獨立參考模型比，不交給「AI 自己說對」。
- **挑戰式測試（R6 落地）** → 拒絕「只會 pass 的測試」。要：負向測試、corner case、**注入已知 bug 驗 testbench 抓不抓得到**（bug injection / mutation）。一個抓不到任何 bug 的 testbench 是裝飾品。
- **三方交叉（R3）** → spec ↔ RTL ↔ TB，任兩者打架就是訊號，先別假設誰對。
- **可重現（R4）** → 記錄 seed、指令、版本；regression 要能重跑復現。
- **AI 容易犯的 DV 錯**（要特別盯）：assertion 寫成永遠為真（vacuous pass）、coverage bin 漏關鍵情境、sequence 沒涵蓋反壓/錯誤注入、checker 跟 DUT 用同一個假設（共模錯誤）。

### 🟦 軟體開發流程

- **AI 寫的 code，讀得懂才收** → 看不懂的程式碼不准進 main（等同收一個 junior 的 PR）。
- **「Show me it running」不是「trust me it works」** → 跑測試、型別檢查、linter、實際執行看行為。
- **新功能必帶測試，且測試要會失敗** → 先確認測試在功能未實作時是紅的（否則是假測試，呼應 DV 的 bug injection）。
- **可重現** → 鎖版本、附指令、附輸入；「在我機器上會動」不算數。
- **把 AI 當 code reviewer 也要驗** → 它說「沒問題」時，要它指出「最可能出事的 3 個地方」。

## 可直接複製的模板

### A. Prompt 輸出契約（貼到任何 AI 任務開頭）
```
請依以下契約輸出：
1. 每個事實型陳述後標 [VERIFIED]/[ESTIMATE]/[STALE]/[ASSUMPTION]，並附來源。
2. 事實與判斷/預測分開，判斷一律標 [ESTIMATE]。
3. 關鍵數字要交叉比對 ≥2 個獨立來源。
4. 開頭給結論摘要，結尾給「我無法驗證的東西」清單。
5. 給出可重現步驟（指令/版本/輸入）。
6. 主動指出你最不確定的 3 點，並論證這結論可能怎麼錯。
不符合契約的輸出視為未完成。
```

### B. AI 產出驗收清單 (Definition of Done)
- [ ] 每個關鍵主張都有來源 + 信心標籤
- [ ] 事實 vs 判斷有分開
- [ ] 關鍵數據 ≥2 來源交叉比對
- [ ] 有「未驗證/未知數」清單
- [ ] 有可重現步驟，我**實際重跑過**至少一項
- [ ] 我**抽樣深查**了 ≥2 個 `[VERIFIED]` 並通過
- [ ] 有對抗審查（矛盾/corner/最弱點）
- [ ] 改動有留痕（改了什麼、為什麼）

### C. DV-specific Review Gate
- [ ] AI 的 assertion 不是 vacuous（有用反例/bug injection 驗過會 fail）
- [ ] coverage 數字看過，關鍵情境的 bin 有覆蓋
- [ ] checker 與 DUT **不共用假設**（避免共模錯誤）
- [ ] testbench 對照 spec 逐條 trace 過
- [ ] regression 可用記錄的 seed 重現

## 反模式 (Anti-patterns) — 看到就退回

1. 接受**沒有來源**的流暢答案（流暢 ≠ 正確）。
2. 信「測過了 / 應該沒問題 / 大概沒事」。
3. 讓 AI **自評**「我做得對嗎」當作驗證（自評不是驗證）。
4. 把**判斷/預測當事實**用。
5. 一次吞一大包、**無法分段檢查**。
6. **不留痕**就更新結論（無法追溯）。
7. 寫**只會 pass** 的測試 / 永遠為真的 assertion。
8. 只看一個來源 / 只跑一個 corner 就下結論。

## 一頁總結（可印出來貼牆上）

> 1. **把 AI 輸出當 DUT** — cover 它、assert 它、scoreboard 它。
> 2. **每句話 = 主張 + 來源 + 信心標籤**，無標籤者不採信。
> 3. **事實與判斷分開**，預測永遠是 estimate。
> 4. **≥2 獨立來源交叉**；IC/DV 走 spec↔RTL↔TB 三方。
> 5. **要可重現的證據**，不要結論；自己重跑、抽樣深查。
> 6. **挑戰不附和** — 找矛盾、注 bug、逼它打自己臉。
> 7. **過程要可見**：流程、糾錯、未知數，全部攤開。
> 8. **不是信任 AI，是讓 AI 可稽核。**
